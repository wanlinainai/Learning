## 场景题

### 如果设计一个秒杀场景，会考虑什么问题？

一个秒杀系统，需要解决的问题有：

1. 瞬时大流量
2. 热点数据
3. 数据量大
4. 库存的正常处理逻辑（超卖、少卖问题）
5. 重复下单
6. 对普通交易是否存在影响

**针对大流量，考虑的就是做流量的过滤和缓存的使用。**

首先一个秒杀的功能，以及前端的访问，这一部分的内容可以考虑放到CDN，用户加载更快。也不会走到我们的服务器。

针对流量来做过滤，很多秒杀系统，可以考虑在前端直接做随机丢弃，筛选一部分幸运买家，返回系统繁忙，之后再试。这种也是可以做到部分流量的平移限制。

之后的话在服务器中需要进行限流处理，比如sentinel，来对接口做限流控制。

之后对于一些高频次的查询操作来做缓存处理，做缓存的话，本地缓存性能要高于分布式缓存。

**针对热点数据，一般考虑的话是做拆分 + 缓存**

将热点数据拆分到多个数据源中，之后通过负载均衡的策略来讲请求拆分到不同的数据源中，可以通过userId来做取余拆分。

秒杀场景是存在提前预知功能的，也就是热点的数据，做缓存是预热即可。

**数据量大**

这种的话考虑的就是查询的效率低的问题。

还是做缓存、要么就用ES，再大的话就需要分库分表，还有就是数据归档，统计时间太长的数据移动到另一个归档库。

**库存的正常处理，主要就是防止 超卖、少卖 的问题**

**重复下单基于Token来做防重即可，在下单请求之前提前获取到一个Token，之后再Redis中存储Token，进行校验即可**

**对普通交易是否存在影响。这种的话基本上就是数据隔离的设计，可以考虑到 逻辑隔离和物理隔离**。逻辑隔离的话就是在商品上打上标签，热点标签。物理隔离就是把数据完全拆分开。

### 如果不用Redis分布式锁，如何防止用户重复点击

最简单的就是在前端做按钮置灰，用户无法点击。

其次的话就是按照Token的机制做幂等，用户访问到页面的时候就会从后端获取到一个Token，下一次点击的时候将Token带过来，服务端对Token进行校验，验证Token是否被使用过，没有使用过的话才可以进行点击。验证的逻辑可以放到数据源中进行的。

基于布隆过滤器过滤是否不存在，不存在一定不存在，来做一次快速失败。

接下来就是对系统的保护相关的，做限流策略，基于滑动窗口来做或者其他方案来做就可以。

### 如果设计一个订单号服务，该怎么做？

唯一性：基于UUID或者雪花算法来做。

数据量：设计订单号的话，需要考虑到后续的数据量如何做兼容，需要提前预留出足够的位数。

可读性：需要明确表名是什么订单号，可以根据业务需求自定义订单号的格式和组成方式，使用时间戳、用户ID等信息来构造订单号。

基因法：最终的话考虑到数据量很大的情况，考虑到分库分表，需要快速路由到对应的库表，比如买家ID。

### 如何防止超卖和少卖问题？

**超卖：就是商品卖多了，但是数据库的库存量不够了。卖了101，但是数据库中只有100个。**

超卖的问题本质上也是高并发的一个问题。核心就是两个点：保证原子性和顺序性。

**先考虑只有数据库层面**：原则上需要保证原子性，只能通过加锁的方式，悲观锁、乐观锁。针对高并发的场景，悲观锁是会出现锁阻塞的问题。很可能将数据库拖垮了。所以考虑乐观锁，加一个版本号，但是这个出现的问题是可能会出现大量失败，同时update本身就是行级别锁，依旧会出现锁阻塞的问题。

那么在库存扣减情况下，不加锁可以实现吗？是可以的。

完全可以依靠数据库本身的执行引擎顺序机制，只要保证库存不要扣减成负数即可。通过SQL即可实现。

```sql
update inventory set quantity = quantity = #{count} where id = '1' and quantity >= #{count}
```

将具体的扣减逻辑交给SQL执行。

这种就可以了。

但是上述的话是完全依靠数据库来抗的，多个线程执行update 还是会出现阻塞，和乐观锁一样了，同时MySQL的行最多也就是只有200 - 300并发更新，如果想要抗的更多的话，只能考虑提升硬件或者其他中间件组合了。Redis就是天然符合的。

依靠Redis的原子性和LUA脚本来实现库存的扣减。之后通过MQ将消息发送到消费者，异步处理数据库库存的扣减。

**少卖：Redis中扣减成功了，但是MQ消息没有发送成功，就会出现少卖的问题。**

少卖的话直接做人工对账即可。少卖的场景本身是可以理解的，原本100个库存，结果最终只卖了99个，剩下了一个可以下一次继续卖啊，对业务影响并不大。

### 如果说处理一个订单的时候出现11：00超时关闭，但是在11:00也支付成功了怎么办？

正常来说，我们在设置支付状态的时候其实是存在支付状态机的，每一步执行流程都会存在明确的流转路径。

一般来说，支付成功和已超时都应该是最终状态。终态之间是不会出现互相流转的过程的。也就是说出现了这种情况的话肯定是状态设计不合理。

**状态流转控制**

比如使用数据库中的update操作来执行状态的更新，update order set state = 'PAY_SUCCESS' where state = 'PAYING' and lock_version = #{lock_version} / update order set state = 'PAY_EXPIRED' where state = 'PAYING' and lock_version = #{lock_version} 。

基于这种的话利用了行锁的特性保证数据库中更新只能更新一条。也就是只有一个终态。

对于支付成功，但是超时失败的话还好操作。超时的update直接拒绝掉。

对于超时成功，但是支付失败的话需要逆向操作，将用户支付的金额**原路退回**。

在进行原路退回的时候需要注意，一定要保证回退成功，支付成功但是超时失败的话需要保证：支付金额 > 0；冲退金额 = 0。超时成功但是支付失败的话需要保证：支付金额 - 冲退金额 = 0。

同时引入定时任务重试。最后留痕做业务对账。

### 多级缓存应用

多级缓存并不只是包含服务端的Caffeine + Redis。在客户端、CDN等都是多级缓存。

其中包含**近端缓存（网络边缘、距离用户近的缓存）**。一个简单的秒杀场景，可能涉及到5层缓存：Client --->  CDN --->  Nginx --->  本地缓存  ---> Redis分布式缓存。

### 如何做数据对账

一般来说就是分成了两种：离线对账 和 实时对账。

实时对账一般指的是准实时，也就是不能保证无延迟，但是一般可以做到秒级别的延迟上，离线对账一般是D+1对账。

> D+1：D指的是自然日，包含工作日和节假日。+1指的是第二天，也就是数据发生变化之后的第二天核对。
>
> T+1：T指的是交易日，工作日，T+1指的就是数据发生之后的下一个工作日进行核对。

对账的实现就是两种：代码对账（定时任务） 和  SQL核对。

但是写代码的方式并不推荐，原因是：数据量变大的话代码实时性很差，并且代码存在执行失败的可能。可能出现OOM的问题。

推荐SQL的方式。

一般来说告警的话是需要及时跟进的，但是有一些是无效告警，可以尝试调整告警的阈值，多次报警之后人工接入。

### MySQL的热点行更新有什么问题？

1. 锁竞争。update本身就是会加排他锁的，导致大量的请求被阻塞。
2. 占用数据库连接。阻塞的线程不会释放连接，一直持有。
3. 消耗CPU资源。大量的锁等待。会导致大量的自旋，多个线程会不断尝试获取锁，CPU不断执行自旋操作，
4. 索引维护的成本大。频繁的更新热点行，不仅会导致数据的变化，还会导致索引相关的维护频繁。
5. 主从延迟增大。

### 一个接口是3000QPS，接口的RT是200MS，预计需要几台机器？

如果接口RT为200ms，我们可以知道单台机器单条线程一秒钟的吞吐量：

1000/200 = 5个请求/s

按照tomcat的默认最大线程200来看。那么单台机器默认就是200 * 5 = 1000个请求。

如果需要3000的QPS，理想状态下需要的机器就是：3台机器。

但是这也只是理想状态，完全忽略了CPU、内存、硬件相关配置，因为随着访问量增大，各种硬件占用情况也是大幅度增大的。

大致预估的话需要在理想情况下按照2-3倍进行估算。

### 一个成熟的电商下单系统，如何设计一个数据一致性方案

现在假设存在几个核心系统：订单系统、库存系统、积分系统、通知系统、链系统。

场景就是订单系统需要创建订单，库存系统需要扣减库存。积分系统可能需要增加积分，如果用的是微信小程序的话可能存在微信通知。

本身来说这就是一个分布式事务的场景，分布式事务存在XA、TCC、Seata等。可靠消息最终一致性（MQ事务消息）、最大努力通知等。

选择的话绝大多数的系统之间都是最终一致性，不是强一致性。原因还是考虑到系统之间的交互尽可能都是成功的，但是可能存在延迟。强一致性本身不适合。但是也不是绝对的。

**订单系统和库存系统采用强一致性**，保证下单成功的话库存一定是扣减成功的。否则就是超卖的问题。库存扣减成功的之后下单也是要成功的，否则就是少卖的问题。

积分系统的话采用的就是最终一致性，不需要保证积分一定要即时的同步成功。延迟也是可以接受的。

通知系统的话是最终一致性，最坏的结果是通知丢失了，但是实际上对业务的影响并不大。可以接受。

订单和链系统的话虽然在业务上来看是强一致性的，但是毕竟是三方系统，他们的可用性保证和本机系统的可用性保证是天差地别的，也是只能尽可能通知成功，如果失败的话也不要紧，做离线同步即可。定时任务，每天凌晨执行一次即可。

### 如果单表的数据量比较大的话，只能考虑分库分表了吗？

也不一定。分库分表的问题其实也很明显，第一个就是要在现有运行中的系统中做分库的话需要修改相当多的代码。同时需要考虑的架构层面的设计也需要非常仔细。

一般还是考虑在现有的数据库上进行修改。

1. 如果是数据量在2000万左右，其实差不多已经到了MySQL的推荐的瓶颈了，优先考虑索引的构建，在代码中的join操作少一点，做数据冗余等。
2. 考虑缓存的使用。考虑本地缓存、分布式缓存来存储常用的数据（一般来说常用的数据量并不会太多）。
3. 做分表。其实就是水平拆分。
4. 做数据归档。定期清理一些老数据，超过一年的数据，放到离线数据库中。

### 实现百万数据导入到数据库

首先考虑其中可能会出现的问题所在：

- 内存溢出：Excel文件数据量太大，全部加载到内存中一定会内存溢出
- 性能问题：数据量太大，单线程处理太慢
- 错误处理：如果在入库的过程中出现各种各样的问题，导致数据存库出现问题

针对内存溢出的问题，一般来说就是基于流式读取的方式进行分批处理。技术选型的话就是基于EasyExcel来做，针对大数据量的Excel文件进行了磁盘的一行行读取数据，逐个解析。

性能问题，单线程的话很慢，如果想要提升性能，需要多线程，使用生产者消费者模式，多个线程读取，多个线程插入，之后做批量插入的功能。

如果出现错误的话，基本上要考虑做定时任务重试，如果是数据重复的话，考虑的就是在数据库层面做唯一性约束了。

### 在做SQL调优的时候，为什么用了主键索引反而导致了查询很慢呢？

```sql
Select * from table_name where delete = 0 and state = 'init' and id > 273482 order by id limit 100
```

这个SQL的查询时间是很长的。我们其实是设置了索引列：delete、state的组合索引的。但是最终还是走的主键索引，没有做到我们的组合索引。

rows行数大概是800多万行。

在走主键索引的时候，id > 273482 的数据依旧有很多，1000多万条，虽然走到了主键索引，但是主键索引的过滤作用不大导致的查询慢的问题。

所以解决方法就是让SQL走索引的话不走主键索引即可。

可以直接强制加上 `force INDEX()`；或者是修改 order by 之后的字段为create_time。

### 一次RPC调用，客户端显示超时，但是服务端并没有超时，可能是什么原因？

这种的话跟服务端就没有什么关系了，主要先看网络连接情况。查看TCP的重传率。比如网络带宽太低，导致TCP重传率增加，或者是在弱网环境导致TCP重传率增加都会出现客户端超时的原因。

第二个的话需要考虑到客户端本身是否存在耗时操作，可能在客户端执行相关逻辑过程中出现大量等待的问题，导致超时。

### 如何做数据迁移

写旧读旧 ---> 双写读旧 ---> 增量数据核对 ---> 存量数据迁移 ---> 存量数据核对 ---> 双写读新 ---> 全量数据核对 ---> 写新读新。

一定要采取的是双写数据，旧库和新库都需要同时写入新的数据。

时刻保持警惕，做旁路验证，对比新库和旧库之间的数据情况，新增加的数据完全一致，因为我们之后是需要将增量数据写入到新库的。如果出现不一致即时人工接入查看数据情况。在处理写数据的时候需要考虑到insert 和 update 两种情况。insert 的话是直接双写的，但是update的话需要判断这个数据是不是增量数据，增量数据的话新旧两个库都需要update，如果是存量数据的话只需要update旧库的数据即可。

接下来就是存量数据的迁移，这个过程的话就是代码驱动了，手写代码将存量数据批量迁移到新的库中。其中主要需要注意的是不要丢失数据。

接下来就是存量数据的核对，做完迁移之后就是需要做切流，将读请求从旧库中切换到新库中。核对的话注意就是将旧库和新库进行每一条的对比。

切流的话也需要考虑到做灰度切流，可以按照userid进行随机分组。美团的话在做灰度是按照cityId的。

最后的话是做切流写新库，旧库完全丢弃，所有的新的写请求写新库数据，旧库彻底放弃。

在做数据迁移的时候也不是完全按照上述这种复杂的设计，如果是离线数据导入的话，那就随便搞了，按照数据量大小，数据量大，做多线程 + 批量处理。如果是平滑迁移的话就需要考虑上述的过程了。

### 为什么秒杀部分不使用分布式锁来实现

秒杀功能最大的问题就是在库存扣减部分，这个部分中需要考虑到的就是：1. 抗更高的并发；2.避免超卖。

我们使用LUA脚本来对付超卖的问题。原因是我们可以在LUA脚本原子性操作中对比库存量是否超量，计算是在LUA中进行的。但是如果加了分布式锁的话计算逻辑是在业务代码中，可能剩下10个，结果扣减了11个，出现超卖的问题。

这是主要的原因。

还有就是使用分布式锁的话需要大量的与Redis进行网络交互，这种情况的话影响了并发的需求。

### 什么是状态机，描述状态机原理

状态机是用于描述一个系统不同状态的转换和行为。包含状态和转换规则。开发中提到的基本都是**有限状态机**，

一个状态机基本包含几个要素：

1. 状态：代表系统可能处于各种状态，比如：“已下单”、“已支付”、“已发货”等。
2. 事件：触发状态流转的事件。比如：”下单“、”支付“、”发货“等。
3. 转换：定义状态之间的转换规则。在某一个事件发生时，系统从一个状态转换到另一个状态的规则。
4. 动作：状态转换发生的时候执行的操作或者行为。

状态机的实现，有很多方式，其中一种就是可以利用状态机的框架，比如：Spring StateMachine。也可以用状态模式自己封装一个状态工具类。



































## DDD领域驱动

DDD的分层架构是什么样子的？从上到下是四层：用户接口层、应用层、领域层、基础设施层。其中DDD分层有着严格的上到调用的关系，只能是从上到下调用，不能逆向调用，比如应用接口层调用应用层以及领域层和基础设施层，应用层调用领域层。

除了上述的简单的DDD4层架构还有洋葱架构和六边形架构。

**洋葱架构：**类似于洋葱一样从外层到内层一层一层。最外层是用于界面（APP、Web服务器和缓存、DB等），往里的话是应用服务，再往里就是领域服务，最核心的是领域模型。

**六边形架构：**最外层是适配器（缓存、DB、APp等），内层是应用层，核心是领域模型。

不论是哪一种架构，最终核心都是领域模型，都是根据领域模型驱动完成对应的业务。
